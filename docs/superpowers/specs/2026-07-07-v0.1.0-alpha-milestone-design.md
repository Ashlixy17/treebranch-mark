# v0.1.0-alpha Milestone Design

## Goals

v0.1.0-alpha is the first release-focused milestone for treebranch-mark.

The goal is to stop adding new visualization modes temporarily and turn the current MVP pipeline into a usable alpha release:

- Generate the first complete browser demo SVG.
- Reduce GitHub API rate-limit friction with optional token support.
- Rewrite the README first screen for fast project understanding.
- Prepare demo evidence and release notes.

This milestone is a product hardening pass, not a new graph algorithm phase.

## Non-Goals

v0.1.0-alpha does not include:

- Timeline Layout
- Metro Layout
- Animation
- Theme system for graph colors
- Hover / Tooltip
- Zoom / Pan
- PNG / GIF export
- GitHub Action
- VS Code extension
- GitLab Source
- Local Git Source

Step 8 should wait until the alpha release is usable and documented.

## Current Baseline

The project currently has a complete MVP rendering pipeline:

```text
GitHubApiSource
      |
      v
GitSourceSnapshot
      |
      v
CommitParser
      |
      v
BranchGraphBuilder
      |
      v
TreeLayout
      |
      v
RenderModelBuilder
      |
      v
SvgRenderer
      |
      v
Browser Viewer
```

The browser page already calls `RenderPipeline` instead of manually wiring Parser, Graph Builder, Layout, RenderModel, and Renderer.

The main practical blocker is anonymous GitHub API rate limiting.

## Release Scope

v0.1.0-alpha should include:

- GitHub Source
- Commit DAG Parser
- Branch Graph Builder
- Tree Layout
- RenderModel
- SVG Renderer
- Browser RenderPipeline
- Optional GitHub Token support
- README first-screen rewrite
- Demo screenshot
- Release notes

## Priority 1: Optional GitHub Token Support

### User Experience

The browser form should add:

```text
GitHub Token (Optional)
```

Behavior:

- Empty token: use anonymous GitHub API.
- Non-empty token: send `Authorization: Bearer <token>` with GitHub REST requests.
- Clearing the field returns to anonymous API.

The token field should make clear that the token is optional.

### Security Constraints

Token support must stay minimal and browser-local:

- Do not save the token to `localStorage`.
- Do not save the token to `sessionStorage`.
- Do not put the token in the URL.
- Do not include the token in `GitSourceSnapshot`.
- Do not include the token in JSON debug output.
- Do not log the token to console.
- Do not store the token in README examples.
- Do not pass the token into Parser, Graph Builder, Layout, RenderModel, or Renderer.

The token only belongs to the GitHub HTTP client boundary.

### Data Flow

Recommended flow:

```text
App.tsx
  |
  v
RenderPipeline
  |
  v
GitHubApiSource
  |
  v
GitHubRestClient
  |
  v
fetch(..., Authorization header)
```

The cleanest MVP API is to configure the Source client at construction time:

```ts
const source = new GitHubApiSource({
  client: new GitHubRestClient({
    token,
  }),
})
```

`GitSourceInput` should not gain a token field. The token is authentication configuration, not repository source data.

### Error Handling

Existing error mapping should remain:

- `not-found`
- `rate-limited`
- `network-error`
- `unsupported-source`
- `unknown`

If a token is invalid and GitHub returns `401`, v0.1.0-alpha can map it to `unknown`. A focused `bad-credentials` error belongs to a future error-handling milestone.

## Priority 2: First Complete Demo

After token support is available, verify at least one complete public repository flow:

```text
Repository: vuejs/core
Branch: main
Action: Generate SVG
Expected: SVG graph appears in browser
```

If the repository is too large or slow for alpha demo, use a smaller public repository:

```text
octocat/Hello-World
Ashlixy17/treebranch-mark
```

The demo should validate:

- Source request succeeds.
- Snapshot summary appears.
- SVG preview appears.
- At least one `<circle>` is rendered.
- For histories with more than one commit, at least one `<line>` is rendered.
- No token appears in debug JSON.

## Priority 3: README First Screen

The README should be reorganized so the first screen answers three questions within seconds:

1. What is this?
2. What does it look like?
3. How do I run it?

Recommended first-screen order:

```text
# treebranch-mark

One-sentence description

Demo screenshot

Quick Start

Architecture

Current Status

Roadmap
```

The detailed layer-by-layer architecture can remain in the README, but it should move below the quick understanding section.

The README must not mention unrelated comparison projects.

## Priority 4: Demo Repositories

Prepare a short list of known public repositories for manual demo testing:

- `octocat/Hello-World`
- `vuejs/core`
- `Ashlixy17/treebranch-mark`

For v0.1.0-alpha, it is acceptable to include one screenshot in README and keep the other repositories as test recommendations.

## Priority 5: Release Notes

Create release notes for `v0.1.0-alpha` with:

### Features

- GitHub Source
- Commit DAG Parser
- Branch Graph Builder
- Tree Layout
- RenderModel
- SVG Renderer
- Browser RenderPipeline
- Optional GitHub Token support

### Known Limitations

- Public GitHub repositories only.
- Browser-only MVP.
- Tree Layout is intentionally simple.
- No Timeline / Metro layout yet.
- No zoom, pan, hover, tooltip, animation, or export.
- Large repositories may be slow because pagination and caching are not yet optimized.

## Testing Strategy

### Unit Tests

Add tests for token support:

- `GitHubRestClient` sends no `Authorization` header when token is missing.
- `GitHubRestClient` sends `Authorization: Bearer <token>` when token exists.
- Empty or whitespace token is ignored.
- Existing error mapping still works.

### Integration Tests

Keep existing `RenderPipeline` tests.

Add or preserve tests that prove:

- Pipeline still returns SVG.
- Source errors still pass through.
- Missing branch heads still return warnings.

### Manual Browser Test

Before release:

```text
npm run dev
Open browser
Input repository
Optional: input GitHub token
Click Generate SVG
Confirm SVG appears
Confirm JSON debug panel does not include token
```

### Release Verification

Before tagging:

```bash
npm test
npm run build
npm run lint
```

## Implementation Sequence

Recommended order:

1. Add optional token support to `GitHubRestClient`.
2. Add token input to browser UI.
3. Wire browser token into `GitHubApiSource` construction.
4. Verify token is not serialized into Snapshot JSON.
5. Capture demo screenshot after GitHub API access succeeds.
6. Rewrite README first screen.
7. Prepare release notes.
8. Tag `v0.1.0-alpha`.

## Definition of Done

v0.1.0-alpha is complete when:

- Browser supports optional GitHub token input.
- Anonymous API still works.
- Token-authenticated API works.
- Token is not persisted or serialized.
- Browser can generate and display an SVG for at least one public repository.
- README first screen contains a concise description, screenshot, quick start, architecture, and roadmap.
- Release notes exist for `v0.1.0-alpha`.
- `npm test` passes.
- `npm run build` passes.
- `npm run lint` passes.
- `main` contains the release-ready code.
- Git tag `v0.1.0-alpha` is created after final verification.
