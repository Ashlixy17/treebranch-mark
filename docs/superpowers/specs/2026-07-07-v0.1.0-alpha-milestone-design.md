# v0.1.0-alpha Milestone Design

## Goals

v0.1.0-alpha is the first release-focused milestone for treebranch-mark.

The goal is to stop adding new visualization modes temporarily and turn the current MVP pipeline into a usable alpha release:

- Generate the first complete browser demo SVG.
- Reduce GitHub API rate-limit friction with optional token support.
- Rewrite the README first screen for fast project understanding.
- Prepare demo evidence and release notes.

This milestone is a product hardening pass, not a new graph algorithm phase.

## Non-Goals

v0.1.0-alpha does not include:

- Timeline Layout
- Metro Layout
- Animation
- Theme system for graph colors
- Hover / Tooltip
- Zoom / Pan
- PNG / GIF export
- GitHub Action
- VS Code extension
- GitLab Source
- Local Git Source

Step 8 should wait until the alpha release is usable and documented.

## Priority 1: Optional GitHub Token Support

### User Experience

The browser form should add:

```text
GitHub Token (Optional)
```

Behavior:

- Empty token: use anonymous GitHub API.
- Non-empty token: send `Authorization: Bearer <token>` with GitHub REST requests.
- Clearing the field returns to anonymous API.

The token field should make clear that the token is optional.

### Security Constraints

Token support must stay minimal and browser-local:

- Do not save the token to `localStorage`.
- Do not save the token to `sessionStorage`.
- Do not put the token in the URL.
- Do not include the token in `GitSourceSnapshot`.
- Do not include the token in JSON debug output.
- Do not log the token to console.
- Do not store the token in README examples.
- Do not pass the token into `GitSourceInput`.
- Do not pass the token into `RenderPipeline.render(input)`.
- Do not include the token in `RenderPipelineResult`.
- Do not add token fields to Parser, Graph Builder, Layout, RenderModel, or Renderer types.

The token only belongs to the GitHub HTTP client boundary.

All security-sensitive constraints involving tokens, cookies, or authentication data must have automated tests. Documentation alone is not enough for these constraints.

### Type Isolation

Token configuration should remain source-specific.

Do not add:

```ts
interface GitSourceInput {
  token?: string
}
```

Recommended shape:

```ts
interface GitHubRestClientOptions {
  token?: string
}
```

`LocalGitSource`, `GitLabSource`, Parser, Graph Builder, Layout, RenderModel, and Renderer should not know that GitHub tokens exist.

### Data Flow

Recommended flow:

```text
App.tsx
  |
  v
GitHubApiSource
  |
  v
GitHubRestClient
  |
  v
fetch(..., Authorization header)
```

The cleanest MVP API is to configure the Source client at construction time:

```ts
const source = new GitHubApiSource({
  client: new GitHubRestClient({
    token,
  }),
})

const pipeline = new RenderPipeline({ source })
```

`GitSourceInput` should not gain a token field. The token is authentication configuration, not repository source data.

`RenderPipeline` should receive a configured `GitSource` dependency, not a token.

### Error Handling

Existing error mapping should remain:

- `not-found`
- `rate-limited`
- `network-error`
- `unsupported-source`
- `bad-credentials`
- `unknown`

If a token is invalid and GitHub returns `401`, v0.1.0-alpha should map it to `bad-credentials`.

The browser UI should display a focused message such as:

```text
Authentication failed. Please verify your GitHub Personal Access Token.
```

This message should be localized with the existing English, Simplified Chinese, and Japanese error-message system.

## Priority 2: Source-Level Memory Snapshot Cache

GitHub API rate limits can block repeated browser demos. v0.1.0-alpha should add a small Source-level memory cache before adding new visualization features.

Recommended boundary:

```text
Browser
      |
      v
RenderPipeline
      |
      v
GitHubApiSource
      |
      v
SourceCache
      |
      v
GitHubRestClient
      |
      v
GitHub API
```

Cache belongs to Source because cache is data acquisition infrastructure.

Pipeline must not know that cache exists.

Parser, Graph Builder, Layout, RenderModel, and Renderer must not know that cache exists.

### Cache Model

The MVP cache stores `GitSourceSnapshot` in memory.

Recommended interface:

```ts
interface SourceCache<TValue> {
  get(key: string): TValue | undefined
  set(key: string, value: TValue): void
  clear(): void
}
```

First implementation:

```ts
class MemorySourceCache<TValue> implements SourceCache<TValue>
```

Recommended TTL:

```text
5 minutes
```

Cache key should include normalized Source parameters:

```text
github:
owner/repo
branch
maxCommitsPerBranch
includePullRequests
includeContributors
includeTags
```

Token must not be part of the cache key. Token authenticates the request; it does not change the public repository snapshot.

HTTP cache with ETag, `If-None-Match`, and `304 Not Modified` is deferred beyond v0.1.0-alpha.

### Source Infrastructure Principle

Source is responsible for data acquisition and infrastructure directly related to data acquisition:

- Authentication
- Cache
- Retry
- Rate limiting
- Request optimization
- Error mapping

Pipeline only consumes `GitSourceSnapshot`.

Pipeline does not care whether data came from network, cache, authenticated request, local Git, GitLab, or a future provider.

## Testing Strategy

### Unit Tests

Add tests for token support:

- `GitHubRestClient` sends no `Authorization` header when token is missing.
- `GitHubRestClient` sends `Authorization: Bearer <token>` when token exists.
- Empty or whitespace token is ignored.
- `GitHubRestClient` maps GitHub `401` responses to `bad-credentials`.
- Existing error mapping still works.
- `GitSourceInput` does not include token fields.
- `RenderPipeline.render(input)` does not accept token fields.
- Source cache hit avoids duplicate GitHub API work.
- Source cache miss loads data and stores the snapshot.
- `cache: 'reload'` bypasses existing cache and refreshes the entry.
- `cache: 'no-store'` bypasses cache reads and writes.
- Cache key does not contain the GitHub token.

### Security Tests

Token leakage protection is P1 for v0.1.0-alpha.

Add an automated test that proves snapshot serialization never includes the token:

```ts
it('does not serialize the github token into the source snapshot', async () => {
  const token = 'ghp_test_token_for_leak_check'
  const result = await pipeline.render(input)
  const json = JSON.stringify(result.snapshot)

  expect(json).not.toContain(token)
  expect(json).not.toContain('ghp_')
})
```

This test should fail if a future change accidentally copies authentication data into `GitSourceSnapshot`.

### Integration Tests

Keep existing `RenderPipeline` tests.

Add or preserve tests that prove:

- Pipeline still returns SVG.
- Source errors still pass through.
- Missing branch heads still return warnings.

### Manual Browser Test

Before release:

```text
npm run dev
Open browser
Input repository
Optional: input GitHub token
Click Generate SVG
Confirm SVG appears
Confirm JSON debug panel does not include token
```

### Release Verification

Before tagging:

```bash
npm test
npm run build
npm run lint
```

## Implementation Sequence

Recommended order:

1. Add optional token support to `GitHubRestClient`.
2. Add token input to browser UI.
3. Add `bad-credentials` error mapping and localized UI message.
4. Wire browser token into `GitHubApiSource` / `GitHubRestClient` construction without adding token to `GitSourceInput` or `RenderPipeline.render(input)`.
5. Add automated token non-serialization tests.
6. Verify token is not serialized into Snapshot JSON.
7. Add Source-level memory snapshot cache.
8. Add cache behavior tests.
9. Create ADR files under `docs/adr/`.
10. Capture demo screenshot at `docs/assets/v0.1.0-alpha-demo.png` after GitHub API access succeeds.
11. Rewrite README first screen.
12. Prepare release notes at `docs/releases/v0.1.0-alpha.md`.
13. Tag `v0.1.0-alpha`.

## Definition of Done

v0.1.0-alpha is complete when:

- Browser supports optional GitHub token input.
- Anonymous API still works.
- Token-authenticated API works.
- Token is not persisted or serialized.
- Token is not accepted by `GitSourceInput`.
- Token is not accepted by `RenderPipeline.render(input)`.
- Token non-serialization is covered by automated tests.
- Invalid token responses show a focused authentication error in the UI.
- Source-level memory cache works and is covered by automated tests.
- Pipeline remains unaware of cache.
- ADR files exist for Source, Parser, LayoutResult, RenderModel, Token Security, and Source Cache Boundary.
- Browser can generate and display an SVG for at least one public repository.
- README first screen contains a concise description, screenshot, quick start, architecture, and roadmap.
- README references `docs/assets/v0.1.0-alpha-demo.png`.
- Release notes exist at `docs/releases/v0.1.0-alpha.md`.
- `npm test` passes.
- `npm run build` passes.
- `npm run lint` passes.
- `main` contains the release-ready code.
- Git tag `v0.1.0-alpha` is created after final verification.
